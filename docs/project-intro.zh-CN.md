# auto-fix-openclaw 中文项目介绍

## 项目定位

`auto-fix-openclaw` 是面向 OpenClaw 网关的生产级自愈框架，核心目标是：

- 让连接问题被持续探测
- 让常见故障被自动修复
- 让升级变化可被安全回放
- 让每次修复过程可审计、可复盘、可运营

一句话：把“靠人抢修”升级为“系统自愈”。

---

## 为什么会做这个项目？（真实动机）

我们在实际使用“龙虾（OpenClaw）”过程中，反复遇到三类问题：

1. **经常断联，影响连续工作**
   - 故障往往不是一次性的，而是会重复出现。
   - 每次都需要人工确认状态、重启、排查日志。

2. **升级后容易出现漂移与断联**
   - 升级带来的配置变化、依赖变化、插件变化，常常导致“旧配置不可用”。
   - 维护策略从“修功能”变成“修升级后遗症”。

3. **需要适配不同 coding 产品/环境**
   - 不同团队工具链不同，维护脚本不可避免地碎片化。
   - 缺乏统一框架会让排障效率随复杂度下降。

这就是 auto-fix-openclaw 的起点：
**把零散经验沉淀成一个稳定、可重复、可持续演进的自动修复系统。**

---

## 核心功能

### 1) 持续健康探测
- 周期执行健康检查与网关状态检查。
- 提前发现退化状态，缩短故障发现时间。

### 2) 分层修复链路（确定性优先）
- 优先执行 restart / doctor / service manager 等确定性动作。
- 确定性失败后，才触发 AI provider 兜底修复（Codex / Claude）。

### 3) 升级后的定制回放（reconcile）
- 版本变化时可自动执行 reconcile。
- 支持 overlay 与配置覆盖回放，减少升级导致的断联。

### 4) 防抖与安全控制
- 冷却、日修复上限、连续失败熔断，防止修复风暴。
- 命令执行模式支持 `safe` 与 `shell`，兼顾安全与历史兼容。

### 5) 可观测性与运营能力
- 每次 attempt 生成完整日志与结果摘要。
- 支持通知通道与 Prometheus 指标输出，便于接入现有运维体系。

---

## 解决了什么痛点？

- **断联恢复慢** -> 自动探测 + 自动修复，显著降低恢复时间。
- **升级后不稳定** -> 通过 reconcile 降低升级引入的不可控风险。
- **跨环境维护成本高** -> 统一修复框架，减少脚本碎片化。
- **人工值守压力大** -> 自动化与审计链路减少重复劳动。

---

## 适合哪些团队

- 有 OpenClaw 网关稳定性要求的团队
- 需要频繁升级但希望降低回归风险的团队
- 希望将排障经验制度化、自动化、可审计化的团队

---

## 项目价值总结

`auto-fix-openclaw` 并不是“又一个脚本”，而是把连接稳定性问题升级成系统工程：

- 从被动抢修 -> 主动探测
- 从单次修复 -> 持续自愈
- 从个人经验 -> 团队可复制能力
